九、一般的性能分析思路


1、网络瓶颈：如带宽，流量等形成的网络环境
2、配置参数：如中间件的基本配置，数据库JDBC连接池等；
3、扩容升级：增加应用服务器，数据库服务器以及客户机的CPU、内存、硬盘等配置
4、SQL优化：如全表扫描、未创建索引、限制结果返回集等；
5、部署架构：如部署负载均衡服务器及负载均衡的策略、分布式集群策略。

十、SQL慢分析

先确定：系统慢还是SQL慢？
思考一下：以下这个SQL有哪里可以提升性能

查询的字段用到了T. *。加大了IO量，无用功，全表查询
在单表查询中，字段ETOR_ID是属于主键，主键是唯一非空值。为何还需要加DISTINCT？

总结SQL 语句响应慢的原因有:

  1. 全表扫描，没有索引或者没有用到索引 —建立增加索引；
  2. I/O 使用率繁忙、吞吐量小，形成了瓶颈效应 — 把数据、日志、索引放到不同的 I/O 设备上，增加读取速度；
    3.使用了“<>” —–就不走索引；(Sqll语句中的不等于)
  3. like’%dd’百分号在前—–不走索引；
  4. not in ,not exist —–不走索引；
  5. 查询出的数据量过大 —-根据查询条件,建立索引,优化索引、限制结果集的数据量 ；（分页取出结果）
  6. 锁或者死锁 —优化 sql 语句；（索引）
  7. 读写竞争资源 —-优化 sql 语句；？
  8. 返回了不必要的行和列 —优化 sql 语句；（查询速度慢）
  9. DISTINCT比GROUP BY要慢—优化 sql 语句；
