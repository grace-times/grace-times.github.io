八、WEB前端优化分析

1. WEB前端优化思考点
  • 哪些因素会影响打开速度？
  • 打开速度如何定位和测试？
  • 显示合理性如何评估和判断？
2. 页面加载时间组成

  • Blocked：查找缓存、等待网络连接可用的时间
  • DNS Lookup：域名解析占用的时间
  • Connection： 连接目标服务器占用的时间
  • Send：发送 HTTP 请求包占用的时间
  • Wait： 发送完请求包，到服务器返回 HTTP 包之间的时间
  • Receive：接收 HTTP 返回包占用的时间
3. 静态内容 VS 动态内容
  • 静态内容
    • 图片、 HTML、Flash、JavaScript文件等
    • 内容不会变、访问速度快！
  • 动态内容
    • 由接口等程序动态生成的内容
    • 内容是由接收到请求时的场景所决定的
  • 所有页面都实现静态化不可能、也有很多问题
    • 大量页面的静态化不现实
    • 页面存在变量或动态变化数据，不适合静态化
4. 压缩

  • 为什么需要压缩
    • 关于用户的上行速度和下行速度
    • HTTP协议的请求和响应字节数据
5. 异步加载
  • AJAX技术
    • 基于XML的异步JavaScript
    • 简单说就是在当前页面不刷新的情况下，实现前后台的交互请求支持的一种异步支持技术
  • 优点
    • 把很多非关键逻辑数据或模块信息通过异步加载的方式分别展示，提高网页加载速度，提升用户体验
    • 可以通过异步加载方式实现页面的滚屏展示，让大数量在用户有需求时再异步拉取，提高前台展现速度和节省带宽成本
  • 缺点
    • 浏览器的部分按钮功能失效（后退）
    • 浏览器（客户端）的处理负载会加大，当网速很慢时，可能会导致浏览器处理出现异常
6. 前端优化原则
  • 尽量减少 HTTP 请求
  • 减少 DNS 查找
  • 用域名划分页面内容
  • 减少DOM元素数量
  • 使iframe数量最少
  • 避免404错误，减少304
7. 合并文件

Image maps（图片地图）：将多个小图片合成一个图片,用map来控制链接

8. 内容优化
  • 减少DNS查找时间
    • 减少域名数量，可以减少DNS查找次数可以节省响应时间
  • 用域名划分页面内容
    • 动静分离
    • 不同域名下的文件可以并行加载
  • 减少DOM元素
    • 对页面构造、渲染的时间都有影响
    • 一个页面的DOM元素越多，Javascript遍历的速度就越慢，前台展示就越慢
  • 减少iframe的数量
    1. Iframe使用优点：解决加载缓慢的第三方内容如图标和广告等加载问题；Security sandbox；并行加载脚本
    2. 缺点：即使内容为空，加载也需要时间；会阻止页面加载
  • 避免404，减少304
    • 404既浪费流量，又影响用户体验
    • 能缓存的内容尽量缓存
9. Javascript和CSS优化
  • 把样式表置于顶部
    • 放在xxx之间，使页面有步骤的加载显示
    • 让页面有序加载可以最大化提升性能
    • 页面底部@import方式最好不要使用
  • 避免使用CSS表达式
    • 一个案例：background-color: expression( (new Date()).getHours()%2 ? “#B8D4FF” : “#F08A00” ）
    • 表达式可能计算频率高，前台性能造成潜在影响
  • 消减Javascript和CSS
    • 对JS和CSS做文件字节大小压缩
10. 性能分析

假如在性能测试需求中我们了解到，有一项指标是要求登录、业务账单查询业务操作的页面响应时间不超过3秒，那么本次测试是否达到了这个要求呢？我们先来看“Average Transaction Response Time（平均事务响应时间图）”

从图形下部我们可以看到，登录和查询业务交易他们的“Average Time（平均响应时间）”分别是9.571秒与0.498秒，从这两个数值来看，账单查询交易业务的事务响应时间0.498秒小于预期的3秒，达到了要求，而登录是9.571秒，大于预期的3秒，不符合要求。

注意：如果页面存在6秒思考时间的话，这样的结果是不完全正确的，因为在统计的登录业务的时候，我们没有去除思考时间，所以，登录功能的实际事务时间应该是9.571秒-6秒=2.571秒，小于预期的3秒，故登录业务的事务响应时间也达到了我们的要求。在平时的性能测试活动中，统计结果的时候需要去掉思考时间，加上思考时间是为了真实的模拟用户环境，统计结果中除去思考时间是为了更真实的反映服务器的处理能力，两者并不矛盾。

Hits per Second（每秒点击数）”反映了客户端每秒钟向服务器端提交的请求数量，如果客户端发出的请求数量越多，与之相对的“Average Throughput (bytes/second)”也应该越大，并且发出的请求越多会对平均事务响应时间造成影响，所以在测试过程中往往将这三者结合起来分析。

在图中显示的是“Hits per Second”与“Average Throughput(bytes/second)”的复合图，从图中可以看出，两种图形的曲线都正常并且基本一致，说明服务器能及时的接受客户端的请求，并能够返回结果。如果“Hits per Second”正常，而“Average Throughput (bytes/second)”不正常，则表示服务器虽然能够接受服务器的请求，但返回结果较慢，可能是程序处理缓慢。如果“Hits per Second”不正常，则说明客户端存在问题，那种问题一般是网络引起的，或者录制的脚本有问题，未能正确的模拟用户的行为。具体问题具体分析，这里仅给出一些建议
